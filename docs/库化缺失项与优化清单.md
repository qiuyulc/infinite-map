# Infinite Map — 库化缺失项与优化清单

本文档用于汇总：在把 `@qiuyulc/infinite-map` 作为一个可被外部团队长期依赖的“通用画布/编辑器库”时，目前仍缺少的关键能力、工程保障与优化点。

> 原则：优先补齐“库化必备”（稳定性、可维护性、可诊断性、可升级），其次再补齐“产品体验增强”。

---

## 1. 近期已暴露/已讨论的问题（Backlog）

### 1.1 交互性能与稳定性
1. **resize 后拖动地图卡顿/闪烁**  
   - 待定位根因：虚拟化进出边界导致的 mount/unmount、camera 更新频率、overlay/HUD 计算量、ResizeObserver 抖动等。
2. **DevTools/Console 打开后拖动卡顿**  
   - 待排查：高频 warning/log、同步布局测量、主线程热点、浏览器 DevTools 额外开销放大等。
3. **HUD/Minimap 等 UI 指针事件被画布 capture 误伤**  
   - 已形成规范方向：所有 UI 根节点统一 `data-im-ui` 标记；持续补齐并回归。

### 1.2 主题/样式层
4. **元素 style 中大量 `--im-*` CSS vars**  
   - 优化方向：base vars 下沉到 CSS（按 theme attribute/class 切换），仅对 override 注入差异 vars；继续检查是否存在节点级重复注入。

### 1.3 可扩展性/可观测性
5. **hooks / 订阅能力不足（宿主侧难以做埋点、约束、联动 UI）**  
6. **性能策略分散（节流、虚拟化策略、move patch 合并等）**  
   - 需要收敛为可配置项，并写入文档，避免在组件内部堆“临时逻辑”。

---

## 2. 作为“一个正常可依赖的库”仍缺少的内容（核心清单）

> 按优先级分组：P0 必须（缺了不敢对外用）、P1 强烈建议（库质量关键）、P2 增强项。

### P0（必须）

#### P0.1 文档模型 / 持久化与版本迁移
- 目标：让外部用户能稳定实现保存/加载/回放/协作，而不是各自私有格式。
- 建议产物：
  - `exportDoc()` / `importDoc()`（至少包含 `nodes`、`camera`、关键配置）
  - `schemaVersion` + `migrations[]`
  - 对外声明：哪些字段是“稳定字段”、哪些是“实验字段”

#### P0.2 更完整的受控模式与状态订阅
- 目标：让宿主接入更直观、更少绕 `ctx.bus/store`。
- 建议补齐：
  - `api.subscribe(event, cb)`（或暴露只读 event bus）
  - 常用 API：`getSelectedNodes()` / `setSelection(ids)` / `getSelectedBBox()` 等
  - 开发期警告：当启用 plugins 但未提供 `onPatches/onNodesChange` 时提示“编辑不会生效”

#### P0.3 测试与回归体系
- 目标：避免每次改动导致交互/历史/分组/虚拟化回归。
- 最小单测覆盖建议：
  - `applyPatchesToNodes` 的正确性
  - history inverse（undo/redo）正确性（含 move/end 合并策略）
  - group/lock/hide 的传递规则与边界情况
  - 选择/框选命中逻辑
  - 虚拟化可见列表（含 keepAlive / panKeepAlive 的策略正确性）

---

### P1（强烈建议）

#### P1.1 错误边界与诊断工具
- 目标：插件出错不拖垮全局，并且能快速定位问题。
- 建议能力：
  - 插件 handler/overlay/command 的错误捕获与上报（不让整个画布挂掉）
  - debug 开关（默认关闭）：输出关键事件序列、耗时统计、可见节点数变化
  - 性能指标：虚拟化计算耗时、overlay 绘制耗时、每帧相机更新频率等

#### P1.2 更明确的 core / ui-kit / demo 边界
- 目标：避免 core API 变得 opinionated，且能让用户按需裁剪。
- 建议：
  - core：不强依赖 UI、默认行为尽量中性
  - ui-kit：toolbar/menu/minimap/rulers/zoomDock/theme/default node 等
  - demo：示例与演示数据只留 playground/示例子入口

#### P1.3 构建与发布规范（工程保障）
- 目标：保证“可发布、可升级、可回滚”。
- 建议：
  - semver + changelog（明确 breaking change）
  - CI：typecheck + build + test
  - exports 子路径的兼容性验证（ESM/CJS/types）

---

### P2（增强项）

#### P2.1 更强的扩展点（策略参数化）
将交互偏好下沉为 options/策略，而不是硬编码规则：
- selection（组钻取、空白点击行为、点击已选行为等）
- marquee（requireShift、命中行为等）
- snapping（是否吸附到视口中心线等）
- group/lock/hide 的语义策略

#### P2.2 协作/多人编辑演进路线
既然已有 patch 模型，可以规划：
- patch 流同步 / 合并策略
- CRDT/OT 的适配点
- 大对象资源（如图表 option）版本化策略与撤销模型

---

## 3. 推荐推进顺序（建议）

1) **P0.1 持久化（doc schema + version + import/export）**  
2) **P0.3 测试（先覆盖最容易回归的核心逻辑）**  
3) **P0.2 受控 API + 订阅能力（降低宿主接入成本）**  
4) **P1.1 诊断/错误边界（提升线上可维护性）**  
5) **P1.2 边界清晰化（core/ui-kit/demo）**  

---

## 4. 备注：hooks/订阅能力的候选清单（摘要）

如果要补 hooks，建议优先以下几类（覆盖面最大）：
- 输入拦截/观测：`onBeforeInput` / `onAfterInput`
- 相机/视口生命周期：`onViewportChange`、`onCameraChange`（可拦截/改写）
- 高频状态：`onSelectionChange`、`onDragLifecycle`
- 错误处理：`onPluginError`（setup/handler/overlay/command）

---

## 5. 功能层：建议内置 vs 可选内置 vs 用户扩展（简略）

### A. 建议库内置（通用、高频、一次实现大家都用）
- 对齐/分布（Align/Distribute）：纯计算 + patches，通用且高频
- 多选缩放/旋转的一致规则：属于基础编辑体验
- 键盘移动（方向键/Shift 大步）+ Esc 退出交互：通用编辑器标配
- 工具模式/交互状态（select/pan/disable）最小闭环：便于宿主统一管理编辑状态

### B. 可选内置（提供插件/模块，默认可不启用）
- 导出 PNG（最常用的导出能力）
- Tab 循环选择（按 zIndex/DOM 顺序）：部分产品需要
- 更完整触控手势：按目标平台取舍

### C. 建议交给用户扩展（强业务/强偏好/维护成本高）
- 节点类型注册（render/hitTest/getBBox 的复杂体系）：库提供挂点即可
- Layer/Tree 面板（UI）：库提供 commands/API，面板形态交给宿主
- SVG/PDF 等高级导出：需求差异大，先交给扩展
