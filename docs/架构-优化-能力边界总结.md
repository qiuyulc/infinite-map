# Infinite Map 项目总结（架构方向 → 优化策略 → 能力边界）

> 面向：项目研发（实现/维护/扩展插件、优化性能、明确职责边界）  
> 范围：`infinite-map-monorepo`（`@qiuyulc/infinite-map` + `@qiuyulc/infinite-map-editor` + `playground/docs`）

---

## 1. 架构方向（Architecture Direction）

### 1.1 分层与职责

项目以“**核心渲染引擎**”与“**编辑器能力**”解耦为第一原则，形成单向依赖：

1) **`@qiuyulc/infinite-map`（Core）**  
   - 目标：提供稳定、可复用的“无限画布/节点渲染/坐标系/视口与相机/插件协议”。  
   - 特征：尽量不携带强业务/强编辑语义的 UI（例如菜单、工具栏、编辑手势集合）。  
   - 提供：
     - 引擎：Engine store（视图/viewport）、高性能 DOM 节点层、背景层等
     - 插件协议：`InfiniteMapPlugin / MapContext / HitTest / Gesture / Store / EventBus`
     - 运行时工具：patch 应用、坐标变换、可视节点计算等

2) **`@qiuyulc/infinite-map-editor`（Editor）**  
   - 目标：提供“编辑器能力集合”和“编辑体验增强（UX）”，以插件化形式组装。  
   - 内容：
     - 编辑交互插件：selection / drag / resize / rotate / marquee / nudge / shortcuts / history / clipboard
     - 编辑 HUD/UI 插件：toolbar / context menu / minimap / rulers / zoom dock
     - 编辑体验增强：例如 hover 高亮层（`createHoverHighlightPlugin`）、吸附辅助线、选框等
   - 原则：editor 插件通过 core 的插件协议与 store/bus/service 工作，不反向依赖 core 以外内容。

3) **`playground`**  
   - 目标：演示/回归/交互验证，承接“默认组装方案”的可视化验证。

> 关键收益：  
> - core 更干净、可作为图库/画布基础设施复用  
> - editor 作为“可替换能力层”，方便按需裁剪（只要你不挂载某插件，就没有该功能/开销）

---

### 1.2 渲染与交互：Engine 优先（性能导向）

整体策略是“**高频状态不走 React render**”，而是由引擎 store + subscribe/rAF 合并更新：

- 视口 transform：通过订阅 store 的 `view.transform` **直接写 DOM style**（避免 React 重绘大量节点）
- 虚拟化：通过可视范围计算 + keepAlive 策略减少 DOM 节点 mount/unmount 成本
- 交互管线：pointer 输入走 HitTest + Gesture（Scheme C），并以 priority/依赖关系控制手势抢占

对应的结果是：
- 节点数量增长时，主要瓶颈从“React render”转移到“命中测试/可视范围计算/DOM 操作与布局”上
- 需要明确：哪些 UI 必须由 React 控制，哪些必须是 imperative/订阅驱动

---

### 1.3 插件系统（Plugin Contract）与“组合式编辑器”

插件 `InfiniteMapPlugin` 的能力项：

- `setup/teardown`：生命周期，注册服务/写 store 初值
- `hitTests`：命中贡献（节点/handle/自定义对象）
- `pointerDownProcessors`：非互斥逻辑（如 selection）
- `gestures`：互斥手势（drag/resize/pan/rotate 等）
- `input`：非 pointer 输入（wheel/key/contextmenu）
- `inputHooks`：输入管线钩子（hitTest 前后、hover change、gesture 前后）
- `overlay` + `slot`：UI 渲染分层（background/overlay/hud）

组合原则：
- **核心交互插件**放 editor 包，并通过 `createDefaultEditorPlugins*` 形成“默认编辑器方案”
- 业务自定义通过“增删插件 + 覆盖 registry items + 覆盖 commands”完成，而不是修改 core

## 2. 优化策略（Optimization Strategies）

### 2.1 核心优化目标

1) 降低 React 重绘频率  
2) 将高频交互（拖拽、平移、缩放）从“state 驱动的 render”转为“store subscribe + rAF”  
3) 命中测试与可视范围计算可控（可缓存/可降级/可局部）
4) overlay 不拖慢主交互（选框、辅助线、minimap 等要避免阻塞）

---

### 2.2 输入与视图（Wheel / Pan / Zoom）

- wheel/trackpad 的高频事件必须 rAF 合并，避免在滚动期产生多次同步 setState
- wheel 的默认行为与插件行为需要可组合：
  - 插件可 `onWheel` 返回 `stop/continue` 控制是否拦截默认缩放/平移
  - UI 容器（toolbar/menu）需要**阻止 wheel 冒泡**，否则滚轮会触发画布缩放/平移（已在菜单/工具栏上处理）

---

### 2.3 拖拽/缩放与 overlay 跟随（Engine 模式）

Engine 模式下，为了性能：
- 节点 drag move 阶段可能只改 DOM transform，而不立即 apply patches（数据不变）

这会带来一个必须处理的 UX 问题：
- selection/resize overlay 不能只靠 nodes 数据刷新，否则会出现“节点动了但选框不动”

对应策略：
- overlay 订阅 `dragState`（或 engine view）变化，以 **imperative transform** 实时跟随
- 同时要处理“容器 resize / overlay rerender”导致的 transform 残留问题（需要在 React render 后重置基线）

---

### 2.4 吸附（Snap）与辅助线（Guides）

吸附的成本来源：
- 计算移动对象 bbox
- 计算参照对象（常为可视节点）的对齐候选
- 画辅助线（overlay 渲染）

策略建议：
- 将“吸附计算”与“辅助线展示”分开开关（已引入 `enabled` 与 `guidesEnabled` 思路）
- 参照集合限制为 `getVisibleNodes()`（符合直觉且可控成本）
- guides 写入做去抖/去重（避免 move 阶段频繁 set store + requestRender）

---

### 2.5 HUD/UI（Toolbar / ContextMenu / Minimap / Rulers）

HUD 的共性优化：
- 不要让 HUD 成为输入事件 target 的“透明大层”（父层 pointerEvents:none，子层按需 auto）
- 长列表 UI（菜单/工具栏）需要尺寸约束与滚动，否则影响布局与可用性（已加入 maxWidth/maxHeight）
- 菜单滚动必须阻止 wheel 冒泡，避免触发地图缩放/平移（已处理）

Minimap 特殊点：
- 需要静态层缓存（节点形状）+ 动态层（视口框/相机）分离
- 外部内容更新但 nodes 不变时，提供 `needsRedraw` 触发重绘

---

### 2.6 虚拟化与 KeepAlive

虚拟化是规模化节点的关键：
- viewport 世界矩形 + overscan 计算可视节点
- `keepAlive(node)` 用于“重组件节点”避免频繁卸载重建
- `panKeepAlive`：在 pan 期间扩大 keepAlive 集合，避免边界抖动导致闪烁

建议：
- 大节点数场景默认启用 virtualization
- 对于“内容昂贵节点”，业务侧务必提供 keepAlive 策略

## 3. 项目能力边界（Capability Boundaries）

### 3.1 明确“项目做什么”

本项目定位：**无限画布上的节点布局与编辑器能力**（类似轻量白板/流程图/图编辑器的基础设施）。

具备：
- 多节点渲染与可视范围管理
- 视图缩放/平移/居中/fit
- 选择、拖拽、缩放、旋转、框选
- 历史记录、剪贴板、对齐分布、层级管理、锁定/隐藏、导出等
- HUD：工具栏/右键菜单/minimap/标尺/缩放 dock
- 插件扩展：自定义手势、命中对象、overlay、命令、输入拦截

---

### 3.2 明确“不做什么 / 不保证什么”

1) **非矢量图形编辑器**  
   - 不提供复杂路径编辑、布尔运算、文字排版系统等（这些属于专业矢量编辑器域）。

2) **非协同系统**  
   - 不自带多人实时协作、冲突合并、CRDT/OT。可由业务在 patches 层实现，但不属于本项目核心承诺。

3) **不保证“无限节点数无限性能”**  
   - 规模上限取决于：节点 DOM 复杂度、可视范围大小、命中测试策略、虚拟化/缓存策略、机器性能。

4) **插件生态仍需要约束**  
   - 插件可写 overlay/hitTest/gesture，若实现不当（高频 setState、全量遍历 nodes 等）会破坏性能模型。

---

### 3.3 约束与注意事项（工程实践）

- UI 容器滚动：需要阻止 wheel 冒泡（否则触发地图缩放/平移）
- overlay 分层：背景效果类建议放 `slot: 'background'`，避免视觉/交互干扰节点
- Engine 模式：任何“看起来应该跟着节点动”的 UI（选框/辅助线/hover）都必须考虑 DOM-only move 情况
- 测试策略：交互类回归建议用 `interactions.test.tsx` 覆盖（如 pan 后 overlay rerender、toggle 开关等）

---

## 4. 建议的后续路线图（可选）

1) 进一步清晰化 editor 专用 keys：将 editor keys 与 core keys 物理拆分，减少 core 暴露面  
2) 为“可滚动 HUD”统一提供一个小的 `StopWheelPropagation` wrapper（避免在多个插件里重复写 onWheelCapture）  
3) 为 overlay 提供更明确的“zIndex 约定/层级 token”，避免相互遮挡问题靠经验修  
4) 为大型节点数场景补充基准测试脚本（FPS、拖拽延迟、hitTest 耗时、可视范围计算耗时）

---
