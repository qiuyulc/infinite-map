# 测试指南（@qiuyulc/infinite-map）

本仓库使用 **Vitest** 作为单元/集成测试框架。

> 目标：覆盖已经实现的核心能力（core/utils/editor plugin 的关键逻辑），并提供最小的交互集成测试，避免回归。

---

## 1. 运行测试

在仓库根目录运行：

```bash
pnpm -C packages/infinite-map test
```

监听模式：

```bash
pnpm -C packages/infinite-map test:watch
```

---

## 2. 测试类型与覆盖范围

### 2.1 单元测试（Node/逻辑为主）
位于：`packages/infinite-map/src/__tests__/*.test.ts`

覆盖示例：
- `applyPatchesToNodes`（patch 应用顺序正确性）
- `groupUtils`（locked/hidden 祖先传播、group 展开）
- `spatialIndex`（候选查询 + 用 `rectIntersects` 做精确过滤）
- `history`（move 合并、undo/redo、add/remove）
- `themeOverrideToCSSVars`（只输出 override 的 vars）

### 2.2 交互集成测试（React + jsdom）
位于：`packages/infinite-map/src/__tests__/*.test.tsx`

覆盖示例：
- 点击选择（SelectionPlugin）
- minimap 点击驱动相机变化（MinimapPlugin）

> 说明：drag/resize/rotate 等“真实拖拽”对 PointerEvent 行为要求更严格。为了保证在 CI/jsdom 下稳定，我们对拖拽逻辑采用 **纯插件单测** 覆盖（见 `dragPlugin.test.ts`），UI 集成测试只覆盖选择/相机等行为。

## 3. jsdom 环境下的必要 mock

Vitest 配置：`packages/infinite-map/vitest.config.ts`
- `environment: 'jsdom'`
- `setupFiles: ['src/__tests__/vitest.setup.ts']`

`vitest.setup.ts` 做了两类关键 mock：

1) **Canvas mock**
- jsdom 不实现 `<canvas>`，因此需要 mock `HTMLCanvasElement.getContext('2d')`
- 这里返回一个最小的 2d ctx（`fillRect/beginPath/...` 等均为 no-op），用于 minimap/高亮层等组件的 effect 正常跑通。

2) **ResizeObserver mock**
- `InfiniteMap` 使用 `ResizeObserver` 监听容器尺寸
- setup 里提供一个最小 mock：`observe(el)` 时读取 `el.getBoundingClientRect()` 并触发一次回调
- 测试用例里通过覆盖 `element.getBoundingClientRect()` 来控制“视口大小”

## 4. 如何新增测试（建议）

### 4.1 新增一个纯逻辑测试（推荐优先）
1. 在 `src/__tests__/` 下新建 `xxx.test.ts`
2. 只测试纯函数/数据结构/patch 生成结果（稳定、快）

### 4.2 新增一个交互测试
1. 在 `src/__tests__/` 下新建 `xxx.test.tsx`
2. 用 `@testing-library/react` 渲染 `InfiniteMap`
3. 关键点：为 `InfiniteMap` 根节点设置非零 bounding rect（否则坐标换算全是 0）

示例（伪代码）：
```ts
const { container } = render(<InfiniteMap ... />);
const root = container.firstElementChild!;
root.getBoundingClientRect = () => ({ left:0, top:0, width:800, height:600, ... } as any);
```
